В главе про [диагностику](diagnostic.md) я показывал ошибки компиляции на примере переопределения переменной:

```bash
error[E0013]: variable 'x' is already defined
  --> src/main.veo:1:5
   |
 1 | let x = 10;
   |     - previous definition was here
 2 | let x = 5;
   |     ^ redefined here
   |
```

Возник ли у вас вопрос: как компилятор понял, где находится имя переменной? Причём имя той переменной, которую он уже давно проверил. По сути имя --- это строка
(`std::string`) и в ней нет данных о позициях. Поэтому я написал специальную структуру `NameObj`. Она нужна, чтобы задавать имена объектам, способным их
хранить (например, тем же переменным). Компилятор хранит в качестве имени не просто строку, а именно `NameObj`, которая уже содержит границы имени (позиции).

```cpp
$#pragma once
$#include <lexer/token.h>
$#include <llvm/Support/SMLoc.h>
$#include <string>
$
$namespace veo::basic {
$
struct NameObj {
    std::string Val;
    llvm::SMLoc Start, End;

    NameObj (std::string val, llvm::SMLoc start, llvm::SMLoc end)
        : Val (std::move (val)), Start (start), End (end) {}

    explicit NameObj (Token tok) : NameObj (std::move (tok.Val), tok.Start, tok.End) {}

    NameObj () = default;

    bool
    operator== (const NameObj &other) const {
        if (this == &other) {
            return true;
        }

        return Val == other.Val && Start == other.Start && End == other.End;
    }

    bool
    operator!= (const NameObj &other) const {
        return !(*this == other);
    }
};
$
$}
```

> [!NOTE]
> В индустриальных компиляторах подобные объекты не хранят строку. Используется пул строк, чтобы хранить уникальные строки один раз.
> Если бы Veo тоже имел такой пул, то `NameObj` хранил бы не `std::string`, а ID из этого пула (`size_t`). Это сильно ускорит копирование и уменьшит
> потребление памяти в разы

Вот визуализация того, что хранит `NameObj`:

```text
let  my_variable  =  10;
     ^          ^
     |          |
     Start      End
```

Пока что не обращайте внимания на конструктор `NameObj (Token)`, потому что мы пока ещё не знаем, что такое токен. О токенах мы узнаем очень скоро, потерпите!
Основываясь на личном опыте, я настоятельно рекомендую для вспомогательных структур сразу определять операторы == и !=. Однажды мне понадобилось сравнить
два вектора структур. Для этого нужно было для самой структуры перегрузить оператор ==. Это было сложно, так как приходилось «рекурсивно» прыгать от одной
структуры к другой и везде перегружать этот оператор.

Вообще, по-хорошему, для качественного вывода ошибок позиции должны иметь все сущности, даже операторы. Я не стал делать этого, чтобы не перегружать код и свой
мозг (я ведь тоже учусь). Для текущей реализации компилятора `NameObj` вполне достаточно.
